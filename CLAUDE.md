## Architecture Overview

`go-cms` is a Go library for creating and verifying CMS/PKCS#7 messages as defined in RFC 5652. Its primary focus is on enabling the use of Ed25519 signatures within this established standard - the first open-source Go library to do so.

### Package Structure

The library has a simple, flat structure:

- **`pkg/cms/`**: Main package containing both signing (`signer.go`) and verification (`verifier.go`) functionality
  - `signer.go`: Creates CMS `SignedData` structures with Ed25519 signatures
  - `verifier.go`: Parses and verifies CMS signatures
  - `errors.go`: Structured error types for validation, signature, and key errors
  - `internal/`: Shared constants and utilities (OIDs, size limits, ASN.1 helpers)
- **`cmd/cms-test-tool/`**: CLI tool for testing CMS operations, used in integration tests

### Key Design Patterns

1. **Standards-Compliant**: The implementation adheres strictly to the relevant RFCs (5652 for CMS, 8410 for Ed25519 OIDs) to ensure interoperability.
2. **Idiomatic Go API**: The library exposes a clean, simple API that feels natural to Go developers, hiding the complexity of ASN.1 encoding and cryptographic operations.
3. **Zero Dependencies**: The core library has no external dependencies beyond the Go standard library and the `golang.org/x/crypto` packages.

### Testing Strategy

The project maintains very high test coverage. Key testing strategies:

- **Round-Trip Tests**: Messages signed by the library are successfully parsed and verified by the library
- **OpenSSL Interoperability**: Signatures generated by `go-cms` can be verified by OpenSSL (see `pkg/cms/signer_test.go`)
- **Docker-based Integration Tests**: `make docker-test` runs full OpenSSL verification in a containerized environment
- **RFC 8032 Test Vectors**: Uses official Ed25519 test vectors to ensure correctness
- **Fuzz Testing**: `pkg/cms/fuzz_test.go` provides fuzzing targets for the verifier

## Critical Implementation Details

### Dual Encoding for SignedAttributes

The core technical challenge is that CMS signatures require **dual encoding** of `SignedAttributes`:

1. **For signing**: Encoded as `SET OF` (tag `0x31`) - what the signature covers
2. **For storage**: Encoded as `[0] IMPLICIT` (tag `0xA0`) - how it's stored in the CMS structure

This is handled by `encodeAttributesAsSet()` and `encodeSignedAttributesImplicit()` in `pkg/cms/signer.go:265-323`.

### Ed25519 with SHA-256

While RFC 8419 recommends SHA-512 for Ed25519, this library uses **SHA-256** for the digest algorithm to maintain OpenSSL compatibility. Ed25519 is a "pure" signature scheme that internally uses SHA-512, so specifying SHA-512 as the digest algorithm causes double-hashing which breaks OpenSSL verification. See `pkg/cms/signer.go:156-162`.

### Certificate Encoding

Certificates are encoded using `[0] IMPLICIT` tagging with raw DER certificate bytes (no nested SET), matching OpenSSL's `CMS_CertificateChoices` expectations. See `pkg/cms/signer.go:428-452`.

## API Surface

The library exposes two main functions in `pkg/cms`:

- `SignData(data []byte, cert *x509.Certificate, privateKey ed25519.PrivateKey) ([]byte, error)`
- `SignDataWithOptions(data []byte, cert *x509.Certificate, privateKey ed25519.PrivateKey, opts SignOptions) ([]byte, error)`
- `Verify(cmsSignature, detachedData []byte, opts VerifyOptions) ([]*x509.Certificate, error)`

All signatures are **detached** (do not contain the original data).
